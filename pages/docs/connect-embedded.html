User guide Twill DB v0.4

Connect as embedded — frameworks

The embedded plane runs the engine inside your process via @twilldb/bun (a typed wrapper over bun:ffi and the C ABI). Queries are function calls — no socket, no round-trip. This page shows the integration shape for the common web frameworks; for the full API surface see Embedded (bun:ffi).

The one rule: open once, share the handle

A Database handle is a long-lived resource. Open it once at module scope and reuse it for every request — do not open per request. Multiple connections to the same file:// URL in one process share one underlying database, so snapshot isolation holds across the app.

// db.ts — a single shared module, imported everywhere.
import { open, type Database } from "@twilldb/bun";

// Opened once when the module is first imported.
export const db: Database = open(process.env.TWILLDB_URL ?? "file://./app.db");

// Create schema on boot (DDL runs in autocommit).
db.exec(`CREATE TABLE IF NOT EXISTS notes (
  id   INTEGER PRIMARY KEY,
  body TEXT NOT NULL,
  ts   TEXT NOT NULL
)`);

Embedded means one process

The embedded handle lives in this process's memory. If you run multiple server instances (or serverless workers) that must share writes, either use a single instance, or switch to server mode over the wire — or point all instances at the same s3:// backend, where the durable single-writer lease coordinates them. Embedded file:// is single-process by design.

Plain Bun (HTTP)

The smallest possible integration — Bun.serve calling the shared handle directly.

import { db } from "./db";

Bun.serve({
  port: 3000,
  fetch(req) {
    const url = new URL(req.url);
    if (url.pathname === "/notes" && req.method === "GET") {
      const rows = db.query("SELECT * FROM notes ORDER BY id DESC LIMIT 50");
      return Response.json(rows);
    }
    return new Response("not found", { status: 404 });
  },
});

Hono

Hono runs on Bun directly, so the engine is just a value you close over in your handlers.

import { Hono } from "hono";
import { db } from "./db";

const app = new Hono();

app.get("/notes", (c) => {
  const rows = db.query("SELECT * FROM notes ORDER BY id DESC LIMIT 50");
  return c.json(rows);
});

app.post("/notes", async (c) => {
  const { body } = await c.req.json<{ body: string }>();
  // Parameter binding is positional with "?" placeholders.
  db.query("INSERT INTO notes (body, ts) VALUES (?, ?)", [body, new Date().toISOString()]);
  return c.json({ ok: true }, 201);
});

export default app; // bun run ./app.ts

Elysia

Same idea — register the handle once and use it inside routes. Elysia's decorate makes it available as ctx.db.

import { Elysia, t } from "elysia";
import { db } from "./db";

new Elysia()
  .decorate("db", db)
  .get("/notes", ({ db }) => db.query("SELECT * FROM notes ORDER BY id DESC LIMIT 50"))
  .post(
    "/notes",
    ({ db, body }) => {
      db.query("INSERT INTO notes (body, ts) VALUES (?, ?)", [body.body, new Date().toISOString()]);
      return { ok: true };
    },
    { body: t.Object({ body: t.String() }) },
  )
  .listen(3000);

Next.js

Next.js can use the embedded engine in the server runtime only (route handlers, server actions, RSC) — never in the browser, and not on the Edge runtime. Use the Node.js runtime and guard against bundler module duplication by stashing the handle on globalThis in development (the same pattern ORMs use).

// lib/db.ts
import { open, type Database } from "@twilldb/bun";

const g = globalThis as unknown as { __twilldb?: Database };
export const db: Database =
  g.__twilldb ?? (g.__twilldb = open(process.env.TWILLDB_URL ?? "file://./app.db"));
// app/api/notes/route.ts
import { db } from "@/lib/db";
import { NextResponse } from "next/server";

export const runtime = "nodejs"; // required: the C ABI needs the Node/Bun runtime

export async function GET() {
  return NextResponse.json(db.query("SELECT * FROM notes ORDER BY id DESC LIMIT 50"));
}

export async function POST(req: Request) {
  const { body } = await req.json();
  db.query("INSERT INTO notes (body, ts) VALUES (?, ?)", [body, new Date().toISOString()]);
  return NextResponse.json({ ok: true }, { status: 201 });
}

Bun vs Node

@twilldb/bun loads the native library through bun:ffi, so the embedded path targets the Bun runtime. Running embedded under plain Node would require a NAPI binding, which is a tracked exploratory item, not a shipped channel — see the roadmap. If you are on Node today, use server mode over pgwire instead.

Transactions & clean shutdown

Wrap multi-statement writes in db.transaction() — it issues BEGIN, runs your function, then COMMIT (blocking until the WAL is durable), or ROLLBACK + rethrow on error.

db.transaction((tx) => {
  tx.query("UPDATE accounts SET balance = balance - ? WHERE id = ?", [100, 1]);
  tx.query("UPDATE accounts SET balance = balance + ? WHERE id = ?", [100, 2]);
}); // commits durably here, or rolls back if either statement throws

// On process shutdown, release the handle.
process.on("beforeExit", () => db.close());

Because the data is just engine rows, this same app can fork its database copy-on-write (Branching) or run disaggregated on object storage (Storage backends) with no code change beyond the connection string.
