User guide Twill DB v0.4

Connect as server — clients & ORMs

Run the engine behind a Postgres-wire listener and anything that speaks Postgres can connect — no Twill-specific driver. This is the integration plane for other languages and for ORMs like Drizzle and Prisma. The rule of thumb: the wire protocol gets a tool connected; whether every query runs depends on the supported SQL subset.

Start the server

One binary serves either backend, chosen by the --db URL.

# Local file backend on port 5433
cargo run -p twill-server -- --listen 127.0.0.1:5433 --db file://./srv.db

# Or disaggregated on object storage
cargo run -p twill-server -- --listen 0.0.0.0:5433 --db s3://my-bucket/mydb

Cleartext, no TLS in this phase

The listener speaks Protocol 3.0 startup with trust auth over cleartext — always connect with sslmode=disable. TLS termination and SCRAM auth are deliberate non-goals for now; terminate TLS at a proxy if you need it on an untrusted network. The full simple and extended query protocols (Parse/Bind/Describe/Execute) are supported, which is what lets prepared-statement clients and ORMs work.

psql & connection strings

Every client uses an ordinary Postgres DSN. The database name is accepted but nominal — the server serves the database given by --db.

psql "postgresql://postgres@127.0.0.1:5433/main?sslmode=disable"

# or component form
psql -h 127.0.0.1 -p 5433 -U postgres -d main
# (set sslmode=disable, e.g. PGSSLMODE=disable)

Node / Bun

Bun ships Bun.sql; on Node use pg (node-postgres). Both work unchanged.

// Bun — built-in Bun.sql
import { SQL } from "bun";
const sql = new SQL("postgresql://postgres@127.0.0.1:5433/main?sslmode=disable");

const rows = await sql`SELECT id, body FROM notes ORDER BY id DESC LIMIT 50`;
await sql`INSERT INTO notes (body, ts) VALUES (${"hello"}, ${new Date().toISOString()})`;
// Node — node-postgres (pg)
import { Client } from "pg";
const client = new Client({
  host: "127.0.0.1", port: 5433, user: "postgres", database: "main", ssl: false,
});
await client.connect();
const { rows } = await client.query("SELECT id, body FROM notes WHERE id = $1", [1]);
await client.end();

Python

psycopg (v3) or any DB-API driver connects the same way.

import psycopg

with psycopg.connect(
    "host=127.0.0.1 port=5433 user=postgres dbname=main sslmode=disable"
) as conn:
    with conn.cursor() as cur:
        cur.execute("SELECT id, body FROM notes WHERE id = %s", (1,))
        print(cur.fetchall())
        cur.execute("INSERT INTO notes (body, ts) VALUES (%s, %s)", ("hi", "2026-06-26"))
    conn.commit()

Go

pgx (or database/sql with lib/pq) connects with a standard URL.

package main

import (
	"context"
	"fmt"
	"github.com/jackc/pgx/v5"
)

func main() {
	conn, err := pgx.Connect(context.Background(),
		"postgres://postgres@127.0.0.1:5433/main?sslmode=disable")
	if err != nil { panic(err) }
	defer conn.Close(context.Background())

	var id int
	var body string
	err = conn.QueryRow(context.Background(),
		"SELECT id, body FROM notes WHERE id = $1", 1).Scan(&id, &body)
	if err != nil { panic(err) }
	fmt.Println(id, body)
}

Drizzle ORM

Use the postgres-js or node-postgres driver and point Drizzle at the listener. Drizzle issues parameterized statements through the extended protocol, which the server supports.

import { drizzle } from "drizzle-orm/postgres-js";
import postgres from "postgres";
import { pgTable, integer, text } from "drizzle-orm/pg-core";

const client = postgres("postgresql://postgres@127.0.0.1:5433/main?sslmode=disable", {
  ssl: false,
  prepare: true,
});
export const db = drizzle(client);

export const notes = pgTable("notes", {
  id: integer("id").primaryKey(),
  body: text("body").notNull(),
});

const rows = await db.select().from(notes).limit(50);

Migrations & introspection

Drizzle's query builder maps cleanly onto the supported SQL. Be cautious with drizzle-kit introspection/push, which leans on full Postgres catalog behaviour — prefer authoring schema with plain CREATE TABLE (via db.execute(...) or migrations you control) and check generated DDL against the SQL reference.

Prisma

Set the postgresql provider and a DATABASE_URL with sslmode=disable.

DATABASE_URL="postgresql://postgres@127.0.0.1:5433/main?sslmode=disable"
// schema.prisma
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model Note {
  id   Int    @id
  body String
}
import { PrismaClient } from "@prisma/client";
const prisma = new PrismaClient();
const notes = await prisma.note.findMany({ take: 50, orderBy: { id: "desc" } });

Prisma expects full Postgres

Prisma's runtime queries (CRUD, filters, pagination) generally map onto the engine, but prisma migrate emits advanced DDL and probes catalog features that exceed the current surface. Manage schema with plain SQL you've validated, point Prisma at the existing tables, and treat unsupported syntax (returned as a clean ENGINE_ERR_SQL) as the boundary — never a silent mis-parse. Track surface growth on the roadmap (Phase 6).

PostgREST & other tools

Tools that auto-generate over a Postgres schema also work: the server answers version probes, reflects tables and foreign-key relationships into the catalog, and rewrites the tool's query templates into engine-runnable SQL. Unmodified PostgREST serves full CRUD plus FK resource embedding over the engine with zero engine changes. For composition specifics see Auth (better-auth) and Analytics (DuckDB).

Connections & scale

The engine is single-writer with one OS thread per connection. For serverless or many-client workloads, put a transaction-mode pooler (PgBouncer / pgcat) in front to absorb connection bursts — see Connection pooling.
