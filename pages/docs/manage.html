Manage a database

The same twilldb binary that scaffolds a project can also talk to a real database: run SQL, inspect the schema, author and apply migrations, generate TypeScript types, load seed data, manage copy-on-write branches, reset and dump a database, and even serve it behind Postgres-wire. Where the scaffolder only writes files, the management commands open a Connection in-process — so they need the engine linked in, behind a cargo feature.

Get the management build

Management links the engine, so it is gated behind a manage cargo feature. The release binaries ship with it on; from a checkout, enable it explicitly:

cargo run -p twilldb-cli --features manage -- <command> …
# or put it on your PATH:
cargo install --path crates/cli --features manage

A default twilldb build stays the lean, dependency-free scaffolder — none of the engine is linked. If you run a management command without the feature, the CLI tells you how to get it.

The connection string picks the transport

Every management command takes a connection string, and the scheme chooses how the CLI reaches the database — exactly like the engine chooses its storage backend:

SchemeTransportUse it for
file:// (and s3://, r2://, gs://)Embedded — opens the engine in this processlocal dev, CI, one-off scripts, an offline/stopped database
postgres://pgwire — connects to a running engine-servera live or shared deployment

The CLI is a writer when it opens a database embedded

Twill is single-writer-per-database, enforced by a durable lease. When the CLI opens file:// it is that writer — so point it at a local or stopped database, not one an app or server currently holds, or the two will contend for the lease. To manage a live deployment, use the postgres:// transport so the server stays the sole writer and the CLI is just a client (see Manage a live deployment below).

Run SQL & inspect

# run one statement or query (table output)
twilldb sql file://./app.db "SELECT id, title FROM notes"

# machine-readable output for scripts
twilldb sql file://./app.db "SELECT count(*) AS n FROM notes" --json
# → [{"n":42}]

# list tables, then describe one (columns, nullability, PK, foreign keys)
twilldb tables   file://./app.db
twilldb describe file://./app.db notes

# an interactive REPL — end statements with ; and use dot-commands
twilldb shell file://./app.db
twilldb> .tables
twilldb> .schema notes
twilldb> SELECT * FROM notes LIMIT 5;
twilldb> .quit

A SELECT prints an aligned table (or JSON with --json); an INSERT/UPDATE/DELETE reports the affected-row count. The REPL accepts multi-line statements and the .tables, .schema [table], .help, and .quit/.exit dot-commands.

Migrations

Migrations are an ordered directory of immutable *.sql files plus a _twilldb_migrations tracking table created in the database on first apply. The model is intentionally close to Supabase / dbmate / golang-migrate.

# create migrations/<timestamp>_add_notes.sql
twilldb migrate new add_notes

# apply every pending migration in order, recording each
twilldb migrate up file://./app.db

# show applied vs pending, and warn if an applied file was edited
twilldb migrate status file://./app.db

Each file is split into statements and applied in order; migrate up records the file's version, name, checksum, and apply time, so a second run is a no-op. If an already-applied file's checksum no longer matches, status and up flag drift — a previously-applied migration was edited. Use --dir <path> to point at a migrations directory other than ./migrations.

A migration is the unit of ordering, not always of atomicity

Twill runs CREATE/ALTER/DROP in autocommit — DDL cannot run inside an explicit transaction. So a DML-only file is applied inside one transaction, but a file containing DDL runs statement-by-statement in autocommit, and a mid-file failure leaves earlier statements applied. Keep one concern per migration (schema changes separate from large data backfills) so a partial failure is easy to reason about. For a risky change, migrate up --branch applies it to a copy-on-write branch first so you can verify before promoting (see Branches & preview-and-swap migrations below).

Branches & preview-and-swap migrations

Branching is Twill's differentiator, and the CLI leans into it. A branch is a zero-copy fork of the database at its committed point — creating one copies no data. Branches are durable on the base, so one invocation creates them and the next sees them:

# fork a copy-on-write branch (an optional label is just for humans)
twilldb branch create file://./app.db feature-x
# → created branch 1 ("feature-x")
#   address it with: file://./app.db#branch=1

twilldb branch list   file://./app.db        # id, parent, fork/head LSN, address
twilldb branch delete file://./app.db 1      # drop the pointer + its private overlay

A branch is addressed by appending #branch=<id> to the base URL, and every other command accepts that address — so you can run SQL, inspect, or migrate against a branch in complete isolation from the base:

twilldb sql    'file://./app.db#branch=1' "INSERT INTO notes (title) VALUES ('draft')"
twilldb schema dump 'file://./app.db#branch=1'
# the base is untouched — writes land only in the branch's private overlay

Preview-and-swap migrations

migrate up --branch [name] turns branching into a migration safety feature. It forks a branch, applies the pending migrations there, and leaves the base untouched — so you can verify a risky change on a zero-copy preview before committing to it. Once satisfied, promote by running the ordinary migrate up on the base (the same migrations apply identically); discard with branch delete. This is safer than an in-place transactional DDL migration, which a single-writer, autocommit-DDL engine cannot offer.

twilldb migrate up file://./app.db --branch try-it   # preview on a fresh branch
twilldb schema dump 'file://./app.db#branch=2'        # inspect the result
twilldb migrate up file://./app.db                    # promote: apply to the base
twilldb branch delete file://./app.db 2               # (or discard the preview)

Generate TypeScript types

gen types reflects the live schema and emits a TypeScript interface per table for the @twilldb/bun client, so your db.query<Row>() calls stay statically typed against the real schema:

twilldb gen types file://./app.db --out src/db-types.ts

Storage classes map to TS as you'd expect: integer/real → number, text → string, blob → Uint8Array, and a vector(N) column → number[]. A nullable column is widened with | null. Without --out the types print to stdout.

Seed data & stats

# run a SQL script (e.g. fixtures or reference data)
twilldb seed file://./app.db seed.sql

# print the engine + storage observability counters
twilldb stats file://./app.db

stats reports the same engine and storage counters the server surfaces over SHOW twill.stats — commits, durable WAL appends, the committed LSN, write-lane contention, and the storage backend's read/cache/fsync counters.

Reset, dump & serve

# reconstruct CREATE TABLE DDL from the live schema (columns, types, PK, FKs)
twilldb schema dump file://./app.db

# rebuild a database from scratch: drop every table → re-apply migrations → seed
twilldb db reset file://./app.db --seed seed.sql --force

# run the engine behind a Postgres-wire listener (wraps engine-server)
twilldb serve file://./app.db --listen 127.0.0.1:5433

schema dump prints the DDL a fresh database would need to match this one's shape — it reflects the catalog, so column types, nullability, primary keys, and foreign keys round-trip (defaults, checks, and standalone unique constraints are not part of the reflected catalog and are not emitted). db reset is destructive, so on a non-empty database it requires --force; it drops every table (including the migration tracking table), re-applies all migrations, and runs the optional --seed script. serve composes the Phase 3 engine-server in-process and blocks, so any Postgres client can connect — the same single-writer rule applies, so the server should be the sole writer of that database.

Branch addresses are for embedded commands only

The <url>#branch=<id> address works with the embedded commands (sql, tables, schema dump, migrate, …). serve runs the base database (the server is the sole writer), so it rejects a #branch= address — branch first, then point a fresh server at the branch overlay if you need to serve one.

Manage a live deployment (postgres://)

When a database is already being served — an engine-server (or a twilldb serve) holding the writer lease — you must not also open it embedded, or the two contend for the lease. Instead point the same commands at the server over postgres://: the server stays the sole writer and the CLI is just a wire client.

# the same commands, just a postgres:// URL — host[:port], default port 5432
twilldb sql      'postgres://user@db.internal:5432/app' "SELECT count(*) FROM orders"
twilldb tables   'postgres://user@db.internal:5432/app'
twilldb describe 'postgres://user@db.internal:5432/app' orders
twilldb schema dump 'postgres://user@db.internal:5432/app'
twilldb gen types   'postgres://user@db.internal:5432/app' --out src/db.d.ts
twilldb migrate up  'postgres://user@db.internal:5432/app' --dir migrations
twilldb stats    'postgres://user@db.internal:5432/app'

The read, inspect, and migrate commands all work over the wire. SQL, seed, and migrations are plain statements the server runs; the inspect commands (tables, describe, gen types, schema dump) reflect the live catalog over a small read-only surface the server exposes, so the output matches the embedded form. The transport is chosen purely by the scheme — nothing else about a command changes.

Branching & serve are embedded-only

Copy-on-write branches are a storage-level feature with no Postgres-wire form, so branch create/list/delete, migrate up --branch, and a #branch= address require an embedded (file://) base — a postgres:// URL is refused with guidance. serve is itself the server, so it too is embedded-only. To preview a migration on a branch of a live database, branch the embedded file while the server is stopped, or fork from a snapshot.
