Storage backends

Twill DB's durable storage is pluggable, and you select it with nothing but the scheme of the connection string. file:// is pure-embedded with zero network; s3:// / r2:// / gs:// is storage-disaggregated and scale-to-zero. The engine, the SQL, the C ABI, and your application code are identical either way — only the URL changes.

The promise

Move from a laptop file to global object storage by changing one string. There is no migration of application logic, no second engine, no different SQL. Durability is WAL-centric in every backend: a commit returns only after its records are durable.

file:// — local, embedded

A crash-safe local database file with zero network. Best for development, single-node apps, offline/edge, tests, and the fastest possible hot path. The WAL is written as CRC-checked, length-prefixed frames and recovers by discarding a torn trailing frame, so every acknowledged commit survives a crash.

import { open } from "@twilldb/bun";
using db = open("file://./app.db");           // relative path
// using db = open("file:///var/lib/twill/app.db"); // absolute path
# Same backend, served over the wire
cargo run -p twill-server -- --listen 127.0.0.1:5433 --db file://./srv.db

Single process

A file:// database is owned by one process. Multiple connections within that process share one database (snapshot isolation holds across handles), but separate processes must not open the same file concurrently. For multi-client access, use server mode or an object-storage backend.

s3:// / r2:// / gs:// — disaggregated

Durability bottoms out on object storage. Internally this backend is an LSM page store plus a compare-and-swap (CAS) commit log: commits are an atomic conditional write, which provides ordered durability and single-writer fencing without any separate coordination service. A local cache keeps hot pages in-process so object-storage latency stays off the read hot path.

using db = open("s3://my-bucket/mydb");
using r2 = open("r2://my-bucket/mydb");
using gs = open("gs://my-bucket/mydb");

What you gain by moving here:

  • Scale-to-zero — compute is stateless, so it can stop when idle and only object-storage bytes bill at rest; the next request warms by replaying the WAL. See Scale-to-zero & lifecycle.
  • Durable single-writer lease — a monotonic CAS epoch fences stale writers, the split-brain guard that makes "single writer" safe across instances.
  • Branching that copies nothing — a branch shares immutable history and writes to a private overlay (a child key-prefix in object storage). See Branching.

Credentials & configuration

Bucket credentials belong in the environment, never in the connection string or in code. Use the standard provider variables for your target.

# AWS S3 / MinIO (S3-compatible)
export AWS_ACCESS_KEY_ID=...
export AWS_SECRET_ACCESS_KEY=...
export AWS_REGION=us-east-1
# For MinIO / custom endpoints, set the endpoint via your deployment config.

# Cloudflare R2
export AWS_ACCESS_KEY_ID=...          # R2 uses S3-style keys
export AWS_SECRET_ACCESS_KEY=...

# Google Cloud Storage
export GOOGLE_APPLICATION_CREDENTIALS=/path/to/service-account.json

Never commit secrets

Connection strings in code, tests, and examples should use file:// only. Real bucket credentials are environment configuration. A secret-scanning hook and CI check exist to catch slips — treat a hit as a stop.

Why R2's zero egress matters here

On a disaggregated engine, a cache miss is a read out of object storage — billed as egress on S3/GCS. A cold cache therefore costs more on those providers. Cloudflare R2 charges zero egress, which removes that cost structurally, making it a natural fit for read-heavy or frequently-cold-started databases. The trade-off is otherwise identical: same engine, same code path.
